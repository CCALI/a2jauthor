Explanation of HotDocs XML answer file structure

(with examples)

 

Basics:

1.      Each HotDocs answer file is essentially a mini-database consisting of one record (the details of one lease transaction, for example).

2.      HotDocs permits users to save information from a document assembly session in two answer file types, one of which is the XML format described here.

3.      HotDocs XML answer files have an "anx" extension in the file name (it may be that the extension does not matter).

4.      HotDocs stores data (answers) in variables. The information may be input by the user during an assembly session or set by scripting in the template.

Storage of answers:

1.      If a HotDocs variable has a value, it is stored in the answer file. If it has no value, it is not stored and there is no reference to it (for repeated variables this is not 100% true—see below).

2.      The XML has one parent node (AnswerSet)

3.      With some exceptions, described below, each HotDocs answer is stored in a single node in the XML. The tag for that node defines the data type. The node that stores the answer is nested inside a node that describes what value is being stored (an attribute of the tag for that node specifies the name of the variable). The latter node in turn is a child of the AnswerSet node.

4.      Here's an example of an single answer in the answer file:

            <Answer name = "Exp_Primary attorney">

                        <TextValue>Mike Elliott</TextValue>

            </Answer>

 

In this instance, "Exp_Primary attorney" is the name of the HotDocs variable (used internally by HotDocs). "Mike Elliott" is the value stored for "Exp_Primary attorney". "TextValue" indicates that this variable and the value stored for it is a text type.

 

5.      There are five different variable data types: Text, True False, Date, Number and Multiple Choice. Each of these is stored in the answer file in a similar fashion, except for multiple choice variables, which are a special case (see below).

6.      Variable types in detail:

a.       As the above example shows, text values are stored simply as literal text.

b.      Date values are stored in the D/M/YYYY format.

Example:

      <Answer name = "Exp_Actual township filing deadline BOR">

                  <DateValue>2/12/2005</DateValue>

      </Answer>

In this instance, the lease end date (variable name is " Exp_Actual township filing deadline BOR ") is December 2, 2005 (note that day comes first in XML). Leading zeros at not used for days or months.

c.       True false values are stored as "true" or "false".

Example:

      <Answer name = "Exp_Complaint filed to Aor CY">

                  <TFValue>true</TFValue>

      </Answer>

 

d.      Number variables are stored as numerals with 7 digits after the decimal.

Example:

      <Answer name = "Exp_AV CY Aor final">

                  <NumValue>164321.0000000</NumValue>

      </Answer>

 

7.      Multiple choice variables are a special case. There are two types of multiple choice variables in HotDocs. One allows selection of only one option from a set. The other allows the user to choose as many options as desired. Both are stored in the answer file in a similar way, but in extracting them you will need to deal with multiple values. Each value stored for a variable is stored in a nested node with a tag labeled "SelValue" and stored as text.

Here's an example of a multiple choice variable with only one option selected:

            <Answer name = "Exp_Fee basis">

                        <MCValue>

                                    <SelValue>Flat</SelValue>

                        </MCValue>

            </Answer>

If the variable permits multiple selections, and those are chosen, then there will be a <SelValue> tag for each answer.

8.      Repeating Variables:

a.       Repeating variables are akin to arrays one encounters in programming languages. Repeating variables can store multiple values under a single variable name. Whether or not a variable repeats is determined by the template developer in HotDocs.

b.      All the values for a particular repeating variable are stored in subnodes of the node for that variable The "RptValue" tag is used to indicate that the inner nodes contain repeating values. Example:

            <Answer name = "Exp_Individual PINs">

                        <RptValue>

                                    <TextValue>10-23-323-037</TextValue>

                                    <TextValue>10-23-323-038</TextValue>

                                    <TextValue>10-23-323-039</TextValue>

                                    <TextValue>10-23-323-040</TextValue>

                                    <TextValue unans = "true"/>

                        </RptValue>

            </Answer>

In this example, there are 4 different PIN numbers. Note the "<TextValue unans = "true"/>". This indicates that there is no value for this variable at this repeat. This tag with the "unans=true" normally appears at the end of the list of repeating values, but it may occur in the middle of the list as well. This is necessary because the answers for repeating variables are usually part of a larger structure, so HotDocs needs to keep track of how many repeats there actually are by leaving a placeholder for unanswered information.

Here's another example (manufactured for this purpose):

            <Answer name = "Exp_Property class">

                              <RptValue>

                                    <TextValue>5-93</TextValue>

                                          <TextValue unans = "true"/>

                                    <TextValue>5-93</TextValue>

                              </RptValue>

           </Answer>

 

In this example, the first and third repeats of "Exp_Property class" have answers, bu the second does not.

c.       There is no way to determine from the answer file alone which variables belong in the same structure (e.g. which ones are part of the tenant address structure).

Miscellaneous Information Regarding Answer Files:

1.      There is no predictable order to the way answers are stored.

2.      HotDocs escapes some characters in the attributes, such as apostrophes (I'm guessing this is an XML requirement.
